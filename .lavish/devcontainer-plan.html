Devcontainer for developing in Zed

A reproducible, containerized dev environment so anyone can open this Axum/Rust workspace in Zed (or any Dev Containers–spec editor) and immediately run just check without installing the toolchain on the host.

Chosen approach: a custom Dockerfile baking the full CI-parity toolset, referenced by devcontainer.json. Single service — no docker-compose (see rationale below).

🎯 Goal

Check out the repo, open in Zed, "Reopen in Container" → a working Rust 1.96 + tooling environment with the HTTP server reachable from the host.

📍 Current state

No .devcontainer/ exists. Toolchain is pinned by rust-toolchain.toml; everything else is installed manually + via CI. Onboarding = host setup.

✅ Desired behavior

A Dockerfile + devcontainer.json that provision the toolchain, dev tools, port forwarding, and editor settings — matching CI so green-locally = green-in-CI.

Decisions (resolved)

Decision A ✓ Custom Dockerfile

Base image & where tools live

Build from mcr…/devcontainers/rust:1-bookworm via a Dockerfile, baking toolchain components + all dev tools into image layers. Fast, deterministic container creates; rebuild only on version bumps.

Bonus: installing components in the Dockerfile against the pinned toolchain removes the need to edit rust-toolchain.toml — the earlier open question is now moot.

Decision B ✓ Full CI parity

Tooling scope inside the container

fmt + clippy + nextest + cargo-deny 0.19.9 + cargo-machete 0.9.2 + just. Every CI job — including the dependency checks — is runnable locally before pushing.

Why no docker-compose yet?

A devcontainer can source its container three ways: a prebuilt image, a build.dockerfile, or a dockerComposeFile. Compose earns its keep only when more than one service must come up together.

Reach for compose when…Today?
App needs a database / cache / queue alongside itout of scope
Multiple containers share a networksingle service
Per-service env files, healthchecks, dependsOn orderingnone yet
One Rust service + its toolchainDockerfile is enough
You're right to ask, and right that it's "not yet." AGENTS.md lists Database/auth/sessions as out of scope, added per-app. The day we add the first backing service (most likely a DB) is the clean trigger to migrate from build.dockerfile → dockerComposeFile. Doing it now would be infrastructure with nothing to orchestrate.

Proposed files

Two files under .devcontainer/. Drafts below — final pin formats verified at implementation time.

.devcontainer/devcontainer.json
{
  "name": "countingsheep",
  "build": { "dockerfile": "Dockerfile" },
  "containerEnv": { "DEV_DOCKER": "1", "PORT": "8888" },
  "forwardPorts": [8888],
  "postCreateCommand": "just --version && cargo --version",
  "mounts": [
    "source=cs-target,target=${containerWorkspaceFolder}/target,type=volume"
  ]
}
.devcontainer/Dockerfile
FROM mcr.microsoft.com/devcontainers/rust:1-bookworm
# Pin toolchain to rust-toolchain.toml + add its components
COPY rust-toolchain.toml /tmp/
RUN cd /tmp && rustup show \
    && rustup component add rustfmt clippy
# Dev tooling — full CI parity
RUN cargo install --locked just cargo-nextest \
    cargo-deny@0.19.9 cargo-machete@0.9.2
Why copy rust-toolchain.toml into the image: it makes rustup show resolve and download exactly 1.96.0 at build time, and lets rustup component add attach rustfmt/clippy to that toolchain — so the editor & CI parity hold without touching repo config.

Remaining risks

Zed Dev Containers support is young. Zed reads the standard devcontainer.json, but feature coverage lags VS Code. Open question: which Zed version are you on, and is its dev-container support enabled? A quick smoke test in the real editor is part of "done."
Image build time. cargo install of nextest/deny/machete compiles from source — the first image build is slow but cached thereafter. --locked keeps it reproducible. Acceptable for a baked image.
Resolved: components-vs-pinned-toolchain — handled inside the Dockerfile, no rust-toolchain.toml edit needed. Scope stays single-service per AGENTS.md.

Implementation steps

  1. Add .devcontainer/Dockerfile (base + components + CI-parity tools).
  2. Add .devcontainer/devcontainer.json referencing it via build.dockerfile.

  4. Add the target/ named-volume mount for build performance.
